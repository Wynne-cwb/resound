# 文档模块 P2（本期形态）：纪要生成纳入关联文档 — 设计 spec

> 日期：2026-06-25　状态：待 review
> 上游：文档模块 P1 已全部落地（[P1 spec](2026-06-25-documents-p1-design.md)）。本 spec 是 P2「生成」的**第一个、最小形态**——录音侧增强纪要。
> 原始诉求：会议一般有对应文档做信息同步，希望文档能辅助 LLM 生成。P2 把「关联文档当背景喂进纪要」落地。

## 1. 范围（locked）

经与用户三轮澄清确定：

| 决策点 | 选择 |
|---|---|
| **核心形态/主入口** | 增强**现有录音侧「会议摘要」**生成（不是独立生成新文档） |
| **文档怎么进 prompt** | **全文注入 + 总字数上限兜底**（超限按边界截断并标注） |
| **可见/可控** | **自动用 + 可见**：有关联文档就自动当背景；可见性复用现有「相关文档」卡 + 摘要区一行轻提示 |

**做**：录音详情「会议摘要」生成时，自动把本场**关联文档全文**作为背景一起喂 LLM；摘要区显示一行「本场 N 篇关联文档已作为背景纳入」。

**不做（各自留后续 spec）**：
- Documents 页独立「生成新文档」（手选多来源 → 产出新 vault 文档）。
- 把 Ask 答案存成文档。
- 富格式（PDF/PPT/docx）、在线源（Google Docs/Notion/URL）。
- 「本次实际用了哪几篇」的持久化（见 §6 取舍）。
- 智能压缩/检索式裁剪文档（超限只做边界截断；检索式是后续可选）。

## 2. 数据流

```
用户在录音 X 的「会议摘要」点「生成 / 重新生成」
  ① 反查 X 的关联文档 → [(title, text=content.md 全文)]
  ② 拼「参考文档」块；累计字数超 maxReferenceDocChars → 截断 + 标注「（文档过长，已截断）」
  ③ 填模板：模板含 {documents} → 用它；不含 → 自动注入到 {transcript} 之前（镜像现有 {transcript} 兜底）
  ④ 走现有摘要管线（Summarizer）生成 → 写 summary.md（schema 不变）
  ⑤ UI：摘要区一行「📄 本场 N 篇关联文档已作为背景纳入」（仅当 N>0），可点跳文档
```

无关联文档时：行为与今天**完全一致**（无 §2① 的块、无 §2⑤ 的提示），零回归。

## 3. 关联文档的反查与读取

- **事实源**：各 `document.yaml` 的 `links: [recording:<id>]`（P1 已定）。
- **反查**：给定 recordingId，找出所有 `linkedRecordingIds` 含它的文档。两条可选路径：
  - vault 扫描：`listDocuments(vaultRoot)` filter（自洽、不依赖 index）。
  - index 镜像：`Index.documentsLinked(toRecording:)`（已有，但只回 id+title，仍要回 vault 读正文）。
- **决策**：新增 Core 自由函数 `linkedDocumentTexts(vaultRoot:recordingId:) -> [(title: String, text: String)]`，内部走 vault 扫描 + `documentContent(dir:)` 读 content.md。App 与（未来）CLI 共用，纯函数好测。

## 4. Summarizer / 模板改动

### 4.1 `{documents}` 占位符
- 模板新增受支持占位符 `{documents}`，与 `{date}{title}{speakers}{transcript}` 并列。
- 填充规则（镜像现有 `{transcript}` 的兜底逻辑）：
  - 模板**含** `{documents}` → 用「参考文档」块替换（无关联文档则替换为空串或省略）。
  - 模板**不含** `{documents}` 但**有**关联文档 → 自动把「参考文档」块注入到 `{transcript}` 之前（与 `{transcript}` 缺省补全同款思路，保证「即开即用」）。
  - 无关联文档 → 完全不注入。

### 4.2 「参考文档」块格式
```
## 参考文档（仅作背景，请优先以会议转录为准）

### <文档标题>（<格式>）
<content.md 全文>

### <文档标题2>（<格式>）
<...>
```
- 顶部一句**消歧提示**：文档是背景、转录是主依据（防 LLM 把文档内容当成会上说过的话）。

### 4.3 字数上限
- `Prompting` 加常量 `maxReferenceDocChars`（建议 ~16000 字，约几 k token；质量>速度取相对宽松值，后续可设可配）。
- 累计多篇文档正文，超上限：在边界处截断当前文档，追加「（文档过长，已截断）」，其后文档不再纳入并在块尾标注「（其余 M 篇文档因长度未纳入）」。

### 4.4 `Summarizer.summarize(...)`
- 加可选参数 `referenceDocs: [(title: String, text: String)] = []`。
- 内部按 §4.1/§4.2/§4.3 组装并填充 `{documents}`。
- 无 referenceDocs 时行为与现状一致。

## 5. App / UI 改动（小，复用现有风格）

- **`LibraryModel` 生成摘要前**：调 `linkedDocumentTexts(vaultRoot:, recordingId:)` 取关联文档文本，传给 `summarize(referenceDocs:)`。摘要生成路径（手动生成 / 重新生成 / 录音入库后自动摘要）统一走它。
- **可见性**：
  - 复用 P1 已有的「相关文档」卡（录音详情 player 与 tab 之间）作为「哪些文档关联本场」的主要可见来源。
  - 摘要区加**一行轻提示**（仅当本场有关联文档时），照现有摘要 meta 文字风格：
    - 已生成的摘要卡顶部：`📄 本场关联的 N 篇文档已作为背景纳入`（点击滚动到上方的「相关文档」卡，由那张卡承载逐篇查看/跳转，避免重复列表）。
    - 未生成的空状态（「生成摘要」按钮旁）：`将纳入本场关联的 N 篇文档作为背景`。
- **不新增开关**：自动使用全部当前关联文档。无关联文档则这些提示都不出现。
- **设计稿说明**：handoff 设计稿无「摘要参考文档」专门元素；本行提示按现有摘要区文字风格直接实现，无需新出图（已与用户确认）。

## 6. 取舍

1. **不持久化「本次实际用了哪几篇」**：摘要区提示由**当前 links** 实时推导。极端情况（生成摘要后又改了关联）提示会与该次摘要实际用的文档轻微不一致——可接受，省一份存储 schema，且 links 变化后重新生成即恢复一致。
2. **全文 + 字数上限**（非检索式裁剪）：会议相关文档（PRD/议程/纪要）多为中等体量，全文最忠实；超限才截断。检索式按需裁剪留作后续可选优化。
3. **复用现有摘要管线/模板/落盘**：不碰 `summary.md` schema、不碰摘要触发链路，仅在「组 prompt」这一步多注入背景，回归面最小。
4. **消歧提示**写进「参考文档」块：降低 LLM 把文档内容误当作「会上说过的话」的风险。

## 7. 验收

- **CLI/单元可验证**：`linkedDocumentTexts` 反查正确；`summarize` 在有/无 referenceDocs、含/不含 `{documents}` 占位符、超/不超字数上限下，组出的 prompt 符合 §4 规则（截断与标注正确）。
- **App 实机**：
  1. 给某录音关联 1~2 篇文档 → 生成摘要 → 摘要应明显体现文档背景（如文档里的目标/术语/决议被纳入），摘要区出现「N 篇文档已纳入」提示。
  2. 无关联文档的录音 → 生成摘要行为与今天一致、无提示（零回归）。
  3. 关联一篇超长文档 → 不报错、被截断并标注。
  4. 模板里手动写 `{documents}` → 文档块出现在指定位置。

## 8. 落地里程碑（详见后续 plan）

- **M1（Core，CLI/单元验证）**：`linkedDocumentTexts` + `Summarizer.summarize(referenceDocs:)` + `{documents}` 占位符 + 字数上限。
- **M2（App 接线）**：`LibraryModel` 生成路径传 referenceDocs + 摘要区可见提示。
- **M3（收尾）**：模板占位符说明同步（Templates 页/文档）、README/data-contract 视情况、STATE/DECISIONS。
